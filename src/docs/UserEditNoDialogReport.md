# UserEditNoDialog.js 仕様書

## 1. 概要

利用者（障害児通所サービス等）の **追加・修正・削除** を行うフォーム画面コンポーネント。
ダイアログではなくページ遷移方式で表示される。

- ファイルパス: `component/Users/UserEditNoDialog.js`
- エクスポート: `UserEditNoDialog`（default + named）、`NextUserDisp`（named）

---

## 2. モード判定

| 条件 | モード | タイトル |
|------|--------|----------|
| URLパラメータにUID有り | 修正モード (`editOn = true`) | 「利用者修正削除」 |
| URLパラメータに `addnew` 含む | 新規追加モード | 「利用者追加」 |

- URLパラメータは `useParams().p` から取得し、数字部分をUIDとして抽出する。

---

## 3. タブ構成

| タブ番号 | 名称 | 内容 |
|----------|------|------|
| 0 | 基本 | 利用者の基本情報・契約情報等 |
| 1 | 口座情報 | 銀行口座情報 |

- `seagull` フラグが有効な場合、口座情報タブは非表示。
- URLに `bankinfo` が含まれている場合は口座情報タブが初期表示される。

## 5. 姓名分割機能（NameInput コンポーネント）

### 5.1 自動分割ロジック
`NameInput` コンポーネント（`StdFormParts.js` 内）にて実装。

- **発動条件**: 初回レンダー時に `lname`（姓フィールド）に値があり、`fname`（名フィールド）が空の場合
- **分割ロジック**: 文字列の中央で分割する
  ```
  分割位置 = Math.floor(文字列長 / 2)
  姓 = 文字列[0 .. 分割位置-1]
  名 = 文字列[分割位置 .. 末尾]
  ```
- **例**: 「山田太郎」→ 姓「山田」名「太郎」（4文字 → 位置2で分割）
- **例**: 「佐藤花子」→ 姓「佐藤」名「花子」
- 分割後「姓名自動分割済」ラベルと左右矢印ボタンが表示される

### 5.2 姓名調整ボタン（handleNameJustify）
自動分割の結果が不正確な場合に、1文字ずつ姓名の境界を調整できる。

| ボタン | 動作 | 制約 |
|--------|------|------|
| → (右矢印) | 姓の末尾1文字を名の先頭へ移動 | 姓が2文字以上の場合のみ |
| ← (左矢印) | 名の先頭1文字を姓の末尾へ移動 | 名が2文字以上の場合のみ |

### 5.3 バリデーション（onBlur時）
| チェック | 条件 | エラーメッセージ |
|----------|------|------------------|
| 禁止文字 | `forbiddenPtn` にマッチ | 「利用できない文字」 |
| 必須チェック | `required=true` かつ空 | 「入力必須項目です。」 |
| かなチェック | `kana=true` かつひらがな以外 | 「かなを入力してください。」 |

- かな入力時はカタカナ→ひらがな自動変換が行われる（`convKanaToHiraAndChk`）。

### 5.4 読み仮名の自動取得（LLM連携）
`UserEditNoDialog` 内の `useEffect` にて実装。

- **発動条件**: 漢字の名前が存在するが対応する読み仮名が空の場合
- **対象フィールド**: `klname`, `kfname`, `pklname`, `pkfname`
- **処理**: `llmApiCall` を呼び出し、漢字→読み仮名を推定
- **LLMプロンプト**: 「日本人の読み仮名を教えてください。最近10年ぐらいで生まれた子どもの名前も考慮して下さい。」
- 取得した読み仮名は `names` ステートに設定され、`NameInput` コンポーネントに自動反映される
- 処理中は「読み仮名取得中」のスナックバーが表示される

### 5.5 保護者名の自動補完
`NameInput` コンポーネント内の `useEffect`（`names` 監視）にて実装。

- 保護者名字（`plname`）が未入力の場合、利用者の名字（`names.lname`）を自動設定
- 保護者のかな名字（`pklname`）が未入力の場合、利用者のかな名字（`names.klname`）を自動設定

## 6. 送信処理 (handleSubmit) とエラーハンドリング

### 6.1 処理フロー

```
1. フォーム値の収集
2. バリデーションチェック（6.2参照）
3. データ加工・補完
4. Redux store 更新
5. API送信
6. 画面遷移
```

### 6.2 バリデーション（エラーチェック順序）

送信時に以下の順序でバリデーションが実行される。エラー発生時は `{ success: false, errMsg }` を返却し、スナックバーで「送信はキャンセルされました。」を表示する。

#### チェック1: 単位名変更チェック
| 項目 | 詳細 |
|------|------|
| 条件 | 単位名(classroom)が変更されており、かつ当月のスケジュール（D2キー）が存在する |
| エラーメッセージ | 「単位名の変更が検出されました。すでに当月の予定実績が設定済みです。一度、予定実績を削除してから変更してください。」 |
| 回避方法 | なし（仮登録チェックでもスキップ不可） |

#### チェック2: 必須項目チェック
| 項目 | 詳細 |
|------|------|
| 条件 | 基本タブ（tab=0）表示時に必須項目が未入力 |
| エラーメッセージ | 「必要な項目が入力されていません。仮登録をするときはチェックをしてから再度送信を行って下さい。」 |
| 回避方法 | 「仮登録」チェックを入れて再送信 |
| 備考 | 口座情報タブ（tab=1）表示時は口座項目の未入力は無視される。口座タブで必須未入力の場合は回避不可 |

#### チェック3: 複数サービス設定エラー
| 項目 | 詳細 |
|------|------|
| 条件 | `.multiServiceError` 要素がDOM上に存在する |
| エラーメッセージ | 「複数サービスの設定を確認してください。単一サービス利用の場合はサービス種別で複数サービスを選択しないでください。」 |
| 回避方法 | なし（仮登録チェックでもスキップ不可） |

#### チェック4: フォームフィールドエラーチェック
| 項目 | 詳細 |
|------|------|
| 条件 | `.MuiFormHelperText-root.Mui-error` 要素がDOM上に存在する |
| エラーメッセージ | 「エラーの箇所をご確認下さい。仮登録をするときはチェックをしてから再度送信を行って下さい。」 |
| 回避方法 | 「仮登録」チェックを入れて再送信（エラー箇所のデータは空文字に置換される） |

#### チェック5: 上限管理整合性チェック (kanriChk)
| 項目 | 詳細 |
|------|------|
| 条件A | 管理/協力事業所が登録済みで管理タイプを外そうとしており、スケジュールに上限管理情報がある |
| メッセージA | 「管理事業所、協力事業所の変更は上限管理の情報を削除してから変更して下さい。」 |
| 回避方法A | なし（先に上限管理情報の削除が必要） |
| 条件B | 管理/協力事業所が登録済みで管理タイプを外そうとしているが、スケジュールに情報なし |
| メッセージB | 「管理事業所または協力事業所が登録済みです。削除する場合はチェックを入れて下さい。」 |
| 回避方法B | 「削除する」チェックを入れて再送信 |

#### チェック6: 同姓同名チェック（新規追加時のみ）
| 項目 | 詳細 |
|------|------|
| 条件 | 新規追加時に既存利用者と同姓同名 |
| エラーメッセージ | 「二重登録しようとしていませんか？」 |
| 回避方法 | 「同姓同名の利用者として送信」ボタンが表示される |

### 6.3 バリデーション回避の仕組み（仮登録チェックボックス）

エラー発生時に表示される仮登録チェックボックスには3種類のラベルがある:

| ラベル | 表示条件 | 効果 |
|--------|----------|------|
| 「仮登録」 | 必須未入力 / フィールドエラー時 | 未入力項目にダミー値を自動補完して登録 |
| 「削除する」 | 上限管理の不整合時（スケジュール無し） | 不要な管理/協力事業所情報を削除して登録 |
| 「それでも変更する」 | （コメントアウト済み、現在未使用） | — |

### 6.4 データ加工処理（バリデーション通過後）

| 処理 | 詳細 |
|------|------|
| 姓名連結 | `lname` + `fname` → `name`、全角スペース除去、特殊文字クリーニング |
| 年齢計算 | `birthday` と `stdDate` から `age`, `ageNdx`, `ageStr` を算出（`ageOffset` 考慮） |
| 半角変換 | `scity_no` を半角に変換 |
| 法人ID付加 | `hid`, `bid`, `stdDate` を付加 |
| 空日付処理 | `contractEnd` が空の場合 `0000-00-00` に設定 |
| 利用停止処理 | `stopUse` が有効なら `endDate` に当月末日を設定、無効なら `0000-00-00` |
| 拡張設定移行 | `over18`, `dokujiJougen`, `dokujiJougenZero`, `sochiseikyuu`, `ageOffset` を `etc` オブジェクト配下に移動。値が空/falsy の場合は `etc` から削除 |
| 複数サービス | `service` に `,` 区切りで格納、各サービスの契約情報を `etc.multiSvc` に格納 |
| 口座情報 | `bankInfoNames` の項目を `etc.bank_info` に移動 |
| 支給量 | `volume` が `0` の場合、原則日数（月日数 - 8）に置換し `volumeStd = true` |
| sindex | 新規追加時、既存最大値 + 10 |
| 医療ケアタイプ | `icareType` が存在する場合 `type` に `,` 区切りで追加 |
| SQLエスケープ | シングルクォートをエスケープ（`escapeSqlQuotes`） |

### 6.5 仮登録時のダミー値補完 (userDatasFillEmpty)

| フィールド | ダミー値 |
|------------|----------|
| `lname`, `fname`, `plname`, `pfname` | `名無し` |
| `klname`, `kfname`, `pklname`, `pkfname` | `ななし` |
| `birthday` | 5年前の1月1日 |
| `hno` | 001〜999の未使用3桁番号 |
| `volume` | `0` |
| `priceLimit` | `4600` |
| `scity_no` | 既存利用者の値 or `012345` |
| `scity` | 既存利用者の値 or `架空自治体` |
| `startDate`, `contractDate` | 当年1月1日 |
| `lineNo` | `1` |
| `brosIndex` | `0`（強制） |
| `pphone` | `045-000-0000` |

複数サービス時の追加補完:
| サフィックス | ダミー値 |
|-------------|----------|
| `-volume` | `10` |
| `-startDate`, `-contractDate` | 当年1月1日 |
| `-lineNo` | `1` |

### 6.6 API送信
- アクション: `Actions.updateUser` に `a: 'sendUserWithEtc'` パラメータ付きでdispatch
- `etc` オブジェクトは `JSON.stringify` でシリアライズして送信
- 送信後 `Actions.sortUsersAsync()` で利用者リストをソート

### 6.7 送信後の処理
- フォームリセット（`formElm.reset()`、DOM存在確認付き）
- `setRecentUser(uids)` で最近編集した利用者を保存
- 新規追加時は `controleMode.appendUser` にタイムスタンプ付きで保存
- `goBack` パラメータがあればその画面へ遷移
- `suppressNavigation` オプション時は画面遷移しない（管理・協力事業所ボタンから呼ばれた場合）
- 戻り値: `{ success: true, hno, uid }` または `{ success: false }`

## 7. 削除処理 (deleteUser)

### 7.1 削除可能条件
以下の**両方**を満たす場合のみ削除ボタンが表示される:
- `thisUser.date === stdDate`（利用者の作成日が当月）
- スケジュール（D2キー）が未登録

### 7.2 2段階確認
1. **1回目クリック**: ボタンラベルが「削除」→「削除実行」に変化、警告メッセージ表示
2. **2回目クリック**: 実際に削除実行
   - Redux store から `Actions.editUser({uid, users, delete: true})` で削除
   - API に `Actions.updateUser({a: 'removeUser'})` で送信

## 8. 利用停止機能

### 8.1 表示条件
- 修正モード（`addnew === false`）の場合に「利用停止」ボタンを表示

### 8.2 動作
- チェックを入れて保存すると `endDate` に当月末日が設定される
- 次月以降はこの利用者が非表示になる
- 当月の利用・請求は通常通り処理される

### 8.3 次月情報との関係
| stopUse | thisUser.next | メッセージ概要 |
|---------|---------------|----------------|
| true | なし | 次月以降は表示されません |
| false | なし | チェックして保存すると利用停止 |
| true | あり | 完全に非表示にはなりません。次月の情報を確認してください |
| false | あり | 同上（チェック後） |

## 9. 次月以降も変更ボタン

### 9.1 表示条件
- `thisUser.next` が存在する
- `stdDate` が先月の初日以降

### 9.2 動作
- `handleSubmit` に `{ updateFuture: 1 }` オプションを渡して送信
- 次月の情報も当月の内容で上書きされる
