# 認証フロー改善ドキュメント

## 1. 改善の背景と目的

### 1.1 改善前の問題点

本システムには2つの認証方式が存在していました：

1. **パスワードログイン**：ユーザーがメールアドレスとパスワードを入力
2. **Cookie認証**：保存された認証キーを使用した自動ログイン

これらの認証方式が異なる実装パスを持っていたため、以下の問題が発生していました：

- **データ取得ロジックの重複**
  - パスワードログイン後：`fetchAll()` でデータ取得
  - Cookie認証後：`makeAcountByKey()` 内で個別にデータ取得
  - 同じデータを取得するにも関わらず、コードが重複

- **Redux action chain の複雑さ**
  - `makeAcountByKey()` 内で複数のAPIコールが連鎖
  - 非同期処理が `.then()` で繋がり、コードの流れが追いづらい
  - 機能追加時に両方のパスを修正する必要がある

- **保守性の低下**
  - データ取得の変更時、複数箇所を修正
  - バグ修正が漏れるリスク

### 1.2 改善の方針

**「認証方式に関わらず、データ取得は `fetchAll` に統一する」**

- async/await を使った読みやすいコード
- データ取得ロジックの一元化
- 既存の `Actions.js` への影響を最小限に抑える

---

## 2. 改善後の認証フロー

### 2.1 パスワード認証フロー

1. **Login.js**
   - ユーザーがメールアドレスとパスワードを入力

2. **dispatch(fetchAccount())** ← Redux thunk
   - API: getAccountByPw でアカウント情報を取得

3. **事業所選択画面表示**
   - ユーザーが事業所を選択

4. **fetchAll()** ← データ一括取得（modules/thunks.js）
   - dispatch(fetchCom()) - 法人情報取得
   - dispatch(fetchTransfer()) - 伝送状態取得
   - dispatch(listUsers()) - ユーザーリスト取得
   - dispatch(fetchCalender()) - カレンダー取得
   - dispatch(fetchSchedule()) - スケジュール取得

### 2.2 Cookie認証フロー（改善後）

1. **Login.js**
   - autoLogin = true （Cookieから判定）

2. **dispatch(makeAcountByKey())** ← Redux thunk
   - stdDateの決定
   - ユーザー設定読み込み

3. **API: getAccountByKey** ← 認証キーでアカウント取得
   - アカウント情報・アカウントリストをstateにセット

4. **fetchAllWithAuth()** ← async関数（modules/thunks.js）
   - API: sertificatAndNew - 認証キー更新
   - 新しいキーをCookieに保存

5. **fetchAll()** ← データ一括取得（パスワード認証と同じ）
   - dispatch(fetchCom()) - 法人情報取得
   - dispatch(fetchTransfer()) - 伝送状態取得
   - dispatch(listUsers()) - ユーザーリスト取得
   - dispatch(fetchCalender()) - カレンダー取得
   - dispatch(fetchSchedule()) - スケジュール取得

## 3. 主要な変更点

### 3.1 新規追加：`fetchAllWithAuth()` (modules/thunks.js)

Cookie認証専用の async 関数を追加しました。

```javascript
export const fetchAllWithAuth = async (prms) => {
  const { dispatch, mail, key, stdDate: inputStdDate } = prms;
  
  // 1. stdDate の決定（Cookieまたは当月1日）
  // 2. ユーザー設定の読み込み（UI設定等）
  // 3. getAccountByKey でアカウント取得
  // 4. アカウント情報を state にセット
  // 5. sertificatAndNew で認証キー更新
  // 6. fetchAll() でデータ取得 ← ここで統一！
}
```

**ポイント**：
- async/await で処理の流れが明確
- エラーハンドリングが容易
- 最終的に `fetchAll()` を呼び出すことでデータ取得を統一

### 3.2 変更：`makeAcountByKey()` (Actions.js)

Cookie認証のエントリーポイントとして残しつつ、内部実装を簡潔化しました。

**変更前**（約170行）：
```javascript
export const makeAcountByKey = (mail, key, stdDate) =>{
  return (dispatch) => {
    // ... 前処理 ...
    axios.post(endPoint(), urlPrms)
    .then(res => {
      // 新しいキーの作成（sertificatAndNew）
      // 法人情報の取得（fetchCom）
      // カレンダーの取得（fetchCalender）
      // スケジュールの取得（fetchSchedule）
      // ユーザーリストの取得（listUsers）
      // ← データ取得が個別に実装されていた
    })
  }
}
```

**変更後**（約90行）：
```javascript
export const makeAcountByKey = (mail, key, stdDate) =>{
  return (dispatch) => {
    // ... 前処理 ...
    axios.post(endPoint(), urlPrms)
    .then(res => {
      // アカウント情報をセット
      dispatch({type: SET_ACOUNT, payload: res.data.dt[0]});
      // アカウントリストをセット
      dispatch({type: SET_STORE, payload: {accountLst}});
      
      // fetchAllWithAuth に委譲 ← シンプル化！
      fetchAllWithAuth({
        stdDate,
        hid: account.hid,
        bid: account.bid,
        dispatch,
        mail: cmail,
        key: ckey,
      });
    })
  }
}
```

### 3.3 変更なし：`fetchAll()` (modules/thunks.js)

既存の `fetchAll()` は変更なし。パスワード認証でもCookie認証でも、最終的にこの関数が呼ばれます。

## 5. メリット

### 5.1 コードの見通し向上

- **データ取得が一箇所に集約**
  - `fetchAll()` を見れば、どのデータが取得されるか一目瞭然
  - 新しいデータ取得を追加する場合、`fetchAll()` に1行追加するだけ

- **async/await で読みやすく**
  - Promise chain（`.then().then()...`）から脱却
  - エラーハンドリングが `try/catch` で統一

### 5.2 保守性の向上

- **変更箇所の削減**
  - データ取得の変更は `fetchAll()` だけを修正
  - パスワード認証とCookie認証の両方に自動的に反映

- **バグ修正が容易**
  - データ取得ロジックが重複していないため、修正漏れが発生しない

### 5.3 拡張性の確保

- **新しいデータ取得の追加が簡単**
  ```javascript
  // fetchAll() に1行追加するだけ
  dispatch(Actions.fetchNewData(sendPrms));
  ```

- **既存コードへの影響が最小限**
  - `Login.js` は変更なし
  - `Actions.js` の他のアクションは変更なし

## 6. 今後の改善案（オプション）

### 6.1 `makeAcountByKey` の完全廃止

現在は後方互換性のために `makeAcountByKey` を残していますが、動作が安定したら以下の対応も検討できます：

```javascript
// Login.js で直接 fetchAllWithAuth を呼び出す
useEffect(()=>{
  if (autoLogin && !cookieAuthAttempted) {
    fetchAllWithAuth({ dispatch });
  }
}, []);
```

これにより、Redux thunk を完全に排除し、すべてを async/await で統一できます。

### 6.2 loadingStatus の活用

現在、`fetchAll()` が呼び出す各アクションの loading 状態を監視することで、すべてのデータ取得完了を検出できます。これを活用して、ローディング画面の表示制御を統一できます。

---

## 7. まとめ

今回の改善により、以下を達成しました：

✅ **認証フローの統一**：パスワード・Cookie認証ともに `fetchAll()` でデータ取得  
✅ **コードの簡潔化**：`makeAcountByKey()` が約170行→90行に削減  
✅ **保守性の向上**：データ取得ロジックの変更が1箇所で完結  
✅ **既存コードへの影響最小化**：`Login.js` や他のアクションは変更なし  

認証方式の違いは入り口だけで、データ取得は共通化されたため、今後の機能追加や保守が大幅に容易になりました。
